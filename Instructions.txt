Ya existen auth, dashboards separados, perfil, suscripciones, billing, pagos, comprobantes y notificaciones.
Cada usuario es único.
No hay organizaciones ni tenants.
Se necesita un canal interno simple para que el usuario pueda solicitar ayuda y el admin pueda dar seguimiento.
No romper módulos actuales.

Stack:
Laravel 12 + Inertia + Vue 3 + JavaScript + Bootstrap 5.3.

Reglas:
- No TypeScript
- Vue con `<script setup>`
- Solo Composition API
- No Options API
- No API separada fuera del flujo web actual
- No librerías nuevas innecesarias
- No refactorizar módulos no relacionados
- Reutilizar estructura visual actual
- Código simple y listo para producción

Objetivo:
Permitir que el usuario `member` cree tickets de soporte y que `admin/superadmin` los gestione desde el panel.

Alcance de esta fase:
- crear ticket
- listar tickets del usuario autenticado
- ver detalle del ticket
- responder ticket internamente
- admin puede cambiar estado y responder
- mantener historial simple de conversación
- notificar cambios importantes si ya existe sistema de notificaciones internas

No incluir todavía:
- chat en tiempo real
- asignación compleja entre agentes
- SLA avanzado
- prioridades automáticas con IA
- adjuntos complejos si no hacen falta
- base de conocimiento pública

Entidades requeridas:

1. SupportTicket
Campos mínimos sugeridos:
- id
- user_id
- subject
- message nullable
- status string
- priority string nullable
- category nullable string
- last_reply_at nullable timestamp
- closed_at nullable timestamp
- created_at
- updated_at

Estados sugeridos:
- open
- pending
- answered
- closed

Prioridades sugeridas:
- low
- medium
- high

2. SupportTicketMessage
Campos mínimos sugeridos:
- id
- support_ticket_id
- user_id nullable
- is_admin boolean default false
- message text
- created_at
- updated_at

Concepto:
- el ticket guarda resumen general
- los mensajes guardan la conversación
- `is_admin` indica si el mensaje fue enviado por admin/staff
- `user_id` identifica al autor cuando aplique

Modelos:
- SupportTicket
- SupportTicketMessage

Relaciones:
- User hasMany SupportTicket
- SupportTicket belongsTo User
- SupportTicket hasMany SupportTicketMessage
- SupportTicketMessage belongsTo SupportTicket
- SupportTicketMessage belongsTo User nullable

Permisos sugeridos:
- support.view
- support.reply
- support.update
- support.close

Member area:
Permitir al usuario final:
- crear ticket
- ver sus tickets
- ver detalle
- responder en tickets abiertos
- no ver tickets de otros usuarios

Rutas member sugeridas:

- GET /member/support/create
- POST /member/support
- GET /member/support/{ticket}
- POST /member/support/{ticket}/reply

Controlador member:

Métodos:
- index()
- create()
- store()
- show()
- reply()

Vistas member:
- resources/js/Pages/Member/Support/Index.vue
- resources/js/Pages/Member/Support/Create.vue
- resources/js/Pages/Member/Support/Show.vue

Campos al crear ticket:
- subject
- category
- priority
- message inicial

Validaciones backend member:
- subject required|string|max:150
- category nullable|string|max:100
- priority nullable|in:low,medium,high
- message required|string|max:5000

Validación reply:
- message required|string|max:5000

Comportamiento member:
- al crear ticket:
  - crear SupportTicket
  - crear primer SupportTicketMessage con mensaje inicial
  - status inicial = open
  - last_reply_at = now()
- al responder:
  - solo en tickets permitidos
  - agregar mensaje
  - actualizar last_reply_at
  - opcionalmente cambiar status a `pending` o `open` según regla elegida

Permitir a admin/superadmin:

- ver detalle completo

- cerrar ticket

Campos editables por admin:

Comportamiento admin:

  - crear SupportTicketMessage con `is_admin = true`
  - actualizar `last_reply_at`
  - opcionalmente poner status = answered
- al cerrar:
  - guardar `closed_at`
  - status = closed

- admin/superadmin ve todos
